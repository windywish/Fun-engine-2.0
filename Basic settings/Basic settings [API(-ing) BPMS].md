 **Objective:** To formally `ADD` a Basic Setting that defines the **Symbiotic Evolution** between **`[micro(-ing) BPMS]`** and **`[macro(-ing) BPMS]`** through **`['universality (class)'(-ing) BPMS]`**.

**Target Axiom/Principle:** `The Principle of Universal Class Inheritance`

**Operation:** `ADD`

**New Definition / Modification Details:**

**Core Essence** This principle asserts that `The API is Scale-Invariant`. It defines **Universal Classes** (Interfaces) that apply equally to a single atom and a galaxy. This allows the system to simulate **Micro** and **Macro** interactions using the same API calls.

**The Mechanism - [The Fractal Interface Pattern]**

- **Definition:** The API defines a Universal Class `IInteractable` with method `interact()`.
- **Micro Implementation:** An `Atom` implements `interact()` as "Bonding."
- **Macro Implementation:** A `Civilization` implements `interact()` as "Diplomacy."
- **Usage:** The Game Engine calls `entity.interact()` without caring if `entity` is an atom or an empire.
- **Symbiosis:** The Micro informs the Macro (Atoms build Empires); the Macro constrains the Micro (Empires control Atoms).

**The Emergent Property - [Infinite Zoom]** The user can zoom from managing a kingdom to managing a blacksmith's inventory without the interface breaking or changing its fundamental logic.

**Alignment with ['Praxial Genesis Canon'(-ing) BPMS]**

- **Layer 1 (Metaphysical):** Aligns with *Fractal Geometry*. Self-similarity at different scales.
- **Layer 2 (Formalism):** Aligns with *Polymorphism* in OOP. Same interface, different behavior.
- **Layer 3 (Phenomenological):** Aligns with *Consistency*. The world follows the same rules everywhere.

**Rationale / Justification:** Simplifies development and allows for "Spore-like" or "Universal" simulations.

**Scope of Application:** Simulation Architecture, UI Design, Input Handling, and Camera Systems.

##### `The Mechanism of Adaptive API Discovery` (**ID:** `API-UNK-015`)

**Objective:** To formally `ADD` a Basic Setting that defines how the **`[API(-ing) BPMS]`** handles the **`['(discover/build) unknown'(-ing) BPMS]`** to be **Robust** and **Resilient**.

**Target Axiom/Principle:** `The Mechanism of Adaptive API Discovery`

**Operation:** `ADD`

**New Definition / Modification Details:**

**Core Essence** This mechanism asserts that `The API Map is not the Territory`. The API acknowledges that there are "Unknown" functions and data types that it has not yet indexed. It includes a mechanism to **Discover** and **Build** these endpoints on the fly when encountered.

**The Mechanism - [The Reflection-Synthesis Loop]**

- **Encounter:** The System encounters a new, unknown object (e.g., a modded item).
- **Reflection:** The API inspects the object's raw data structure (Reflection).
- **Synthesis:** It dynamically generates a new API wrapper for this object, mapping its raw data to standard system interfaces.
- **Adaptation:** The System can now interact with this "Unknown" object as if it were a native part of the code.
- **Symbiosis:** The Unknown provides Novelty; the API provides Compatibility.

**The Emergent Property - [Mod Resilience]** The game doesn't crash when it sees unknown data. It adapts. It wraps the unknown in a "Safe Interface" and lets the user play with it.

**Alignment with ['Praxial Genesis Canon'(-ing) BPMS]**

- **Layer 1 (Metaphysical):** Aligns with *Epistemological Openness*. We are always ready to learn new things.
- **Layer 2 (Formalism):** Aligns with *Reflection* and *Dynamic Proxy* patterns.
- **Layer 3 (Phenomenological):** Aligns with *Curiosity*. "What is this? Let's find out."

**Rationale / Justification:** Essential for Modding support and "Open World" games where the content is not fully known at compile time.

**Scope of Application:** Mod Loading, DLC Integration, Procedural Content, and Error Handling.
